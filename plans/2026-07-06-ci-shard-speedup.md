## Problem (confirmed, evidence-cited)

`.github/workflows/ci.yml`: **12 jobs**, all `runs-on: ubuntu-latest`, all triggered on **both** `push` and `pull_request` (no branch filter), no `needs:`, no `concurrency:`, no `timeout-minutes`, no caching. Observed durations (prior PR runs, not re-verifiable in-tree ‚Äî treat as direction): `db-verify` ~13min, `verify` ~12min, `qualification-selftest` ~4.5min; rest are seconds.

**Redundancies**
- `verify` (ci.yml:20) and `db-verify` (ci.yml:55) run the **identical** `./scripts/verify.sh`; `db-verify` is a strict superset (adds `RUN_DB_SMOKE=1` + `AUDIT_PURGE_TEST_DB_ISOLATED=1` + DB env + the postgis service). The non-DB body (lint verify.sh:108-113, mypy :115-120, the qualification+authority gate scripts :70-95) runs **twice**. `RUN_DB_SMOKE=1` un-skips the DB-gated tests (**per-test `skipif` across ~23 files**, not per-file) in the same pytest run ‚Äî that full-suite-against-real-Postgres is **intentional integration coverage**, not waste. Only lint + mypy + the qualification/authority validators are true duplication.
- The qualification suite runs **three times** (verify, db-verify, qualification-selftest).
- `run_provenance_check.sh` runs in both `supply-chain` (:95) and `dependency-attestations` (:121) ‚Äî but this is NOT worth de-duping (see Dropped).
- `push` + `pull_request` both fire ‚Üí all 12 jobs run **twice** per same-repo PR-branch push. No `concurrency` cancel ‚Üí stacked pushes pile up.

**Inefficiencies**: zero pip cache, zero mypy `.mypy_cache`, uncached `apt-get install postgresql-client` on db-verify (:46), uncached `docker build` (:167), no concurrency-cancel.

**Inconsistencies**: `verify` (:8) + `db-verify` (:22) omit a `permissions` block while the other 10 pin `contents: read`; `python-version '3.12'` hardcoded in 12 places.

## Vetted plan (ordered; both passes' adjustments folded in)

Expected win: eliminate the 2√ó double-fire (rec 2) + stacked-run pile-up (rec 1) ‚Üí the dominant reduction in **runs launched**; caching + rec-5 dedup shave the long poles. Framed as **wall-clock + queue-slot + cache-budget**, not "compute cost" (public repo = free minutes). Wall-clock on the green path is unchanged-to-slightly-better ‚Äî the two long poles stay parallel (NO `needs:` serialization).

1. **Concurrency cancel, main-exempted** (near-zero risk). `concurrency: {group: ${{ github.workflow }}-${{ github.ref }}, cancel-in-progress: ${{ github.ref != 'refs/heads/main' }}}`. Main exemption mandatory: repo is public (attestations at :137-151 run) and main is unprotected (a cancelled main run is never re-required ‚Üí orphans that commit's CI+attestation evidence AND its rec-3/4 cache-populating save). NOTE: rec 1 alone does NOT de-dup the push+PR pair (they group on different refs ‚Äî `refs/heads/<branch>` vs `refs/pull/N/merge`); it only cancels stacked pushes to the same ref. Double-fire elimination comes from rec 2.
2. **Trigger de-dup + `workflow_dispatch`** (largest win). `on:` ‚Üí `push: {branches: [main]}` + `pull_request:` + `workflow_dispatch:`. CONDITION: this removes automatic CI from the ~35 PR-less dual-agent worktree branches. `workflow_dispatch` only works once the dispatch-bearing ci.yml is on main, and requires someone to run `gh workflow run ci.yml --ref <branch>`. **Write this convention into `state/agent-inbox` for BOTH lanes as part of this slice** (not after), and confirm the Codex lane acknowledges it.
3. **pip caching, explicit path, pilot first** (MODIFY). On the `pip install -e backend[dev]` steps add setup-python `cache: 'pip'` **with `cache-dependency-path: backend/pyproject.toml`** (only `backend/requirements-prod.lock` exists; bare `cache: pip` WARNS-then-degenerates to an empty-hash key, not a hard error ‚Äî so the pilot must check the computed key in the run log). Dev deps are unpinned `>=` (pyproject:21-30), so the pyproject-hash key stays stable while resolvable wheels drift (with dependabot on `/backend`, cache serves stale wheels until pyproject changes) ‚Äî acceptable. **Pilot on `verify` ONLY**, confirm a non-empty pyproject-derived key + a cache save on a main warm-up push, then replicate. This is also the mechanism that cheapens rec 6's kept install.
4. **mypy `.mypy_cache` via actions/cache** (~1‚Äì2min, intermittent). Key on source hash + mypy/python version + restore-keys prefix fallback. Save on main pushes so PR branches restore. Lands with the cache recs (3/4) **before** rec 5. (Corrected: the first-pass "sequence AFTER rec 5" note contradicted the 4-before-5 landing order; 4 lands before 5.)
5. **verify vs db-verify dedup ‚Äî guard lint/type/validators, parallel, NO `needs:`** (MODIFY strongly; biggest blast radius; lands LAST). Reject `needs:[verify]` (serializes ~25min vs ~13min parallel). `test_release_readiness_artifacts.py:56-68,190-205` pins the db-verify run string `./scripts/verify.sh` + its 3 env keys ‚Äî setting `CI_DB_SLICE_ONLY` in the existing env block is safe (asserts presence of the 3 existing keys only); do NOT change the run string or step count.
